Why Observability Skills Are the New Must-Have for Engineers

January 3, 2025 6 min read

As AI takes over more code generation, the ability to understand and debug running systems becomes the differentiating skill. Observability isn't just for SREs anymore - it's becoming a core competency for every engineer.

The Shift in Engineering Work

Ten years ago, most engineering time went into writing code. Today, more time goes into understanding, debugging, and maintaining systems. AI accelerates this trend - you can generate more code faster, but someone still needs to verify it works.

This shift makes observability skills essential. Logs, traces, and metrics aren't just for ops teams during incidents - they're how engineers understand what their code actually does in production.

The Three Pillars

Modern observability rests on three pillars, and engineers need fluency in all of them:

  • Logs: Structured records of events. Understanding how to write useful logs and query them efficiently.
  • Traces: Following requests across distributed services. Invaluable for debugging microservices.
  • Metrics: Aggregate measurements over time. Essential for understanding system behavior and capacity.

Why AI Makes This More Important

When AI generates code, you lose some of the intimate understanding that comes from writing every line. You might not know exactly how a function handles edge cases until you see it fail in production.

Observability bridges this gap. Good instrumentation lets you understand AI-generated code's behavior without reading every line. You verify through observation rather than inspection.

Skills to Develop

Engineers looking to strengthen their observability skills should focus on:

  • Query languages: Learn to write effective queries in tools like Datadog, Splunk, or Grafana
  • Instrumentation: Know how to add useful logging and tracing to code
  • Dashboard design: Create visualizations that surface problems quickly
  • Alert design: Set thresholds that catch real issues without alert fatigue
  • Debugging workflows: Develop systematic approaches to investigating issues

The Interview Gap

Traditional interviews rarely test observability skills. Candidates write code on whiteboards but never debug it with logs. They design systems but don't explain how they'd monitor them.

This is a significant blind spot. Engineers who can't debug production systems effectively become bottlenecks, no matter how well they perform in algorithm interviews.

How We Evaluate This

At Xebot, our debugging challenges provide candidates with real logs, traces, and metrics. We watch how they:

  • Form hypotheses about root causes
  • Navigate observability tools efficiently
  • Correlate information across different signals
  • Communicate their findings clearly

These skills predict on-the-job effectiveness far better than algorithm puzzles.

Getting Started

If you want to build observability skills, start with your current projects:

  • Add structured logging to a service you own
  • Set up distributed tracing across two services
  • Build a dashboard that shows your service's key metrics
  • Practice investigating issues using only observability tools

The engineers who master these skills will be the ones who can ship reliable software in an AI-assisted world.
